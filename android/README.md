# kayaknav — Android

Android port of kayaknav. This crate is a thin shim: it installs the
Android logger bridge, picks a cache path under the app's internal data
directory, and hands the `AndroidApp` off to `kayaknav::launch_android`.
All the actual app code — map, trip table, waypoint logic, NOAA
harmonic store — lives in the parent crate and is shared with the
desktop and wasm builds.

## Prerequisites

1. **Android SDK + NDK**
   - Install via Android Studio's SDK Manager, or standalone
     command-line tools.
   - Required: SDK Platform 34, NDK 26.x+, Build-Tools 34.x.
   - Export `ANDROID_HOME` (e.g. `~/Android/Sdk`) and `ANDROID_NDK_HOME`
     (e.g. `$ANDROID_HOME/ndk/26.2.11394342`). Gradle reads both.

2. **Rust targets**
   ```bash
   rustup target add aarch64-linux-android
   # Add `x86_64-linux-android` if you plan to run on the Android
   # emulator (Intel/AMD host). Legacy `armv7-linux-androideabi` /
   # `i686-linux-android` are only needed for 32-bit hardware.
   ```
   The `cargo.targets` list in `app/build.gradle.kts` controls which
   ABIs the plugin cross-compiles on every build — keep it in sync with
   the triples you've installed via `rustup`, or you'll see `can't find
   crate for core` errors during `:app:cargoBuildX86_64` (etc).

3. **cargo-ndk** (invoked by `rust-android-gradle`)
   ```bash
   cargo install cargo-ndk
   ```

4. **JDK 17**. Any distribution works; `sdkman` or the system package
   manager is fine. Required by Gradle 8.x.

5. **Gradle 8.10.1 via wrapper**. The `gradle-wrapper.properties` in
   this project pins Gradle to 8.10.1. **Do not run `gradle` directly**
   from your system install if it's version 9 or above — the
   `rust-android-gradle` 0.9.6 plugin uses `setFileMode(Integer)` which
   Gradle 9.0 removed, and you'll hit:

   ```
   Unable to find method '… AbstractCopyTask … setFileMode(java.lang.Integer)'
   ```

   Use `./gradlew` (which downloads 8.10.1 transparently) or install
   Gradle 8.x alongside 9.x and invoke that explicitly.

## Build + install

Everything is driven from Gradle; the plugin handles the Rust
cross-compile automatically.

```bash
# From this directory (android/):
./gradlew :app:assembleDebug

# APK lands at: app/build/outputs/apk/debug/app-debug.apk
adb install -r app/build/outputs/apk/debug/app-debug.apk
adb shell am start -n net.avrutin.kayaknav/android.app.NativeActivity
adb logcat -s kayaknav              # filter to our log tag
```

For a release build add `-PcargoProfile=release` — this switches the
Rust profile; Gradle's release signing still needs a real keystore for
Play Store publishing.

First-time Gradle setup: if there's no `gradlew` script in the
directory yet, materialize the wrapper first. We ship only
`gradle/wrapper/gradle-wrapper.properties` (pinning the Gradle version);
the jar + launcher scripts are regenerated by Gradle itself:

```bash
# Uses whatever `gradle` is on your PATH (can be 9.x — that's fine,
# because all this step does is download the 8.10.1 jar referenced in
# gradle-wrapper.properties and write out ./gradlew). Every invocation
# after this point goes through the pinned wrapper, not your system
# gradle.
gradle wrapper
```

Alternatively, opening the project in Android Studio once creates the
wrapper automatically (and also generates `local.properties` with the
SDK path).

### After hitting `setFileMode(Integer)`

If you already ran `gradle` directly before reading this and hit the
`Unable to find method … setFileMode(Integer)` error, clear Gradle's
cached build state before retrying so it doesn't replay the failed
Gradle 9 invocation:

```bash
rm -rf .gradle build app/build
./gradlew :app:assembleDebug
```

## What's wired up

- `src/lib.rs` — `android_main` entry point. Bridges `log` to logcat,
  sets `Config::tile_cache_dir` to `{internal_data_path}/tile_cache`,
  runs `kayaknav::setup::build` via `block_on`, and then hands off to
  `kayaknav::launch_android(bundle, app)`.
- `Cargo.toml` — depends on `kayaknav = { path = ".." }` and the
  Android-only glue (android-activity + android_logger + futures).
  All UI/rendering deps are transitive through kayaknav.
- `app/build.gradle.kts` — AGP 8 + `rust-android-gradle` plugin wiring
  `cargoBuild` into the APK packaging pipeline for every ABI listed
  under `cargo.targets`.
- `AndroidManifest.xml` — uses `android.app.NativeActivity` so no Java
  or Kotlin shim is required; the Rust .so is the whole app.

## How the feature-flag split works

The parent `Cargo.toml` splits the `galileo` dep into per-target blocks
so each platform gets the minimum feature set it needs — but
`fontconfig-dlopen` is **on everywhere non-wasm**, including Android.

That last bit is counterintuitive, so it's worth explaining. font-kit
(pulled in transitively by galileo) has an unconditional dep on
`yeslogic-fontconfig-sys` on every unix-family target, Android
included. Its feature flag `source-fontconfig-dlopen` (which galileo
re-exports as `fontconfig-dlopen`) does NOT remove that dep — it only
changes fontconfig-sys's build.rs from "run pkg-config" to "compile
dlopen stubs." Turning the feature off on Android therefore doesn't
shed any weight; it just makes `cargo build --target
aarch64-linux-android` fail with "pkg-config has not been configured to
support cross-compilation." With the feature on, compilation works;
`dlopen("libfontconfig.so")` will fail at runtime on Android, but
font-kit's dlopen path handles that gracefully (the fontconfig font
source just reports zero fonts).

Android system fonts are still reachable via `/system/fonts/` — they're
not wired in by default, but
`galileo::render::text::TextService::instance().load_fonts("/system/fonts/")`
can be called from `android_main` if on-map labels are needed.

## Future work

- **Soft keyboard / IME support.** Currently we use `NativeActivity`,
  which doesn't give us a native text input path. If the Android UI
  grows text entry (station search, waypoint names), switch to
  `game-activity` — it has better IME wiring but requires a small
  Kotlin `MainActivity` that subclasses `GameActivity`. eframe has a
  `game-activity` feature flag that lights up the corresponding winit
  path.

- **Tile cache size limit.** `.tile_cache` has no LRU cap on desktop
  either, but on a phone it matters more. `RasterTileLayerBuilder`
  doesn't expose a size cap; adding one upstream is the right fix.

- **Jetpack Compose UI.** Everything here assumes the Rust `cdylib`
  owns the UI. To move to a Compose UI layer, switch `NativeActivity`
  to a Kotlin `ComponentActivity` that hosts Compose, drop
  `eframe`/`egui`/`winit` from the transitive dep set, and expose
  kayaknav's Rust logic (trip calc, sweep, best-departures export)
  over JNI or UniFFI. The `scheduling::Trip` API is already
  CLI-friendly (see `src/bin/kayaknav_trip.rs`), so the bridge layer
  is small. Map rendering would still use galileo+wgpu driven into a
  `SurfaceView` that Compose embeds via `AndroidView`.
